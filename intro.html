테스트가 정말 버그를 잡는지 측정합니다.

통과하는 테스트가 의미 있는 테스트인지는 다른 문제입니다. test-validity-evaluator는 코드에 변이를 주입하고 에이전트로 테스트를 적대적으로 흔들어, 어떤 테스트가 비어 있는지 단일 보고서로 알려줍니다.

변이와 적대, 두 방향에서 동시에.

Mutation은 소스 코드를, Adversarial은 테스트 코드를 흔듭니다. 서로 겹치지 않는 각도의 약점을 잡아내고, 단일 ValidityReport로 합쳐 보고합니다.

Track 01 · Mutation
i.

LLM-based Semantic Mutation Testing

AST 기반 syntactic 변이 6종과 LLM 기반 semantic 변이를 결합합니다. 살아남은 변이는 곧 "잡지 못한 결함의 좌표"이며, survivor-diagnose 프롬프트가 보강해야 할 어서션까지 자연어로 제시합니다.

// mutation_orchestrate.py ROR user.age >= 18 → user.age > 18 killed by test_grant_at_boundary COR flags.a && flags.b → flags.a || flags.b SURVIVED ← 어서션 누락 diagnosis: assert_called_with(flag_b=True) 미존재 [score] 0.76 → target 0.85 (9 survivors)
  • 016종 syntactic operator. AOR · ROR · COR · LCR · BoundaryShift · ConstantReplace.
  • 02semantic-operator 프롬프트. 도메인 의미를 보존한 채 결함 패턴 기반 변이를 생성합니다.
  • 03equivalence-judge 자동 필터. 동작이 동일한 변이를 사람 확인 없이 제거합니다.
  • 04MutationLedger 전 이력 보존. 생성·필터·실행·진단의 모든 단계가 한 파일에 남습니다.

Track 02 · Adversarial
ii.

Agentic Adversarial Testing

5종 공격 프롬프트(boundary · edge-case · fuzzer · spec-divergence · state)가 테스트 자체를 적대적으로 검사합니다. 동어반복 어서션·잘못된 결합도·이름과 동작의 불일치를 critique 프롬프트 4종이 별도로 짚어냅니다.

// adversarial_orchestrate.py case test_transfer_negative_amount probe amount = -0.0001 (boundary) → PASS 서비스가 BadRequest 반환 case test_username_validation probe critique-tautology → FAIL assert "a" in "abc" 형식의 무력 어서션 — finding f_8c12e3
  • 015 adversarial 생성자. 경계값·엣지·퍼저·스펙 괴리·상태 의존 시나리오를 분담합니다.
  • 024 critique 평가자. assertion·behavior-naming·coupling·tautology 약점을 분리해 보고합니다.
  • 03AdversarialLedger. 각 case 의 생성자·근거·결과를 mutation과 동일한 형식으로 기록합니다.
  • 04샌드박스 실행 강제. 어댑터의 run.py 안에서만 동작 — 프로덕션 경로 차단.

02 단일 오케스트레이터, 네 단계

한 번 발화 → 끝까지 자동.

orchestrate.py가 분류·매핑·평가·집계를 순차 실행하고, 6개의 종료 조건이 보강 루프를 무한히 돌지 않도록 막습니다.

01

Tier Classification

e2e·integration·flaky를 걸러내 unit 테스트만 평가 대상으로 남깁니다. classify.py가 included/excluded를 출력합니다.

02

Subject Location

어댑터의 resolve.py가 각 테스트의 SUT를 추적해 TestSubjectMap을 만듭니다. 이후 모든 단계의 공통 입력.

03

Mutation + Adversarial

두 트랙이 같은 SubjectMap 위에서 병렬로 돌고, 각자 ledger·findings를 ValidityReport에 합칩니다.

04

Aggregation

dedupe·우선순위·클러스터링 후 ValidityReport JSON과 자체 포함 HTML 한 쌍이 out/<session>/iter-N/ 에 떨어집니다.

03 설치 한 번, 어느 레포에서나

글로벌 symlink로 한 번만.

install.sh가 ~/.claude/skills/test-validity-evaluator로 symlink를 걸어둡니다. 이후 모든 레포의 Claude Code 세션이 자동 디스커버 — git pull 만으로 업데이트.

Quickstart
3 lines
# 1. 글로벌 설치 (한 번) git clone https://github.com/binary-ho/test-test.git chennai cd chennai && ./install.sh # 2. 어느 레포에서나 — Claude Code 세션에서 자연 발화 "이 프로젝트 테스트 품질 평가해줘" "테스트가 진짜 버그를 잡는지 확인해줘" # 3. 결과는 cwd 기준 out/<session>/iter-N/ 에 out/evs_2026-05-18_a1b2/iter-0/validity_report.json out/evs_2026-05-18_a1b2/iter-0/report.html

한 번의 발화로, 테스트의 빈자리가 드러납니다.
