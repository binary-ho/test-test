Test Validity Evaluator
개요 Mutation Adversarial 통합 흐름 사용법 상태
ADR-1 · ADR-2 기반

테스트가 진짜로 잡고 있는지, 두 축으로 검증합니다

테스트 작성 에이전트가 만든 테스트는 "통과한다"는 신호만으로 품질을 보증할 수 없습니다. 두 개의 Claude Skill이 서로 다른 방향에서 테스트를 흔들어, 약점과 누락을 actionable한 finding으로 돌려줍니다.

Skill A · 민감도

코드를 흔든다

원본 비즈니스 로직에 결정론적·의미 변이를 주입해, 테스트가 변이를 잡는지 측정합니다.

Skill B · 견고성

테스트를 흔든다

약한 assertion·tautology를 정적 critique로, 경계·상태·fuzz·명세 괴리를 동적 공격으로 드러냅니다.

개요

두 스킬은 동일한 오케스트레이터(test-validity-evaluator) 아래에서 호출되며, 공통 디스커버리 산출물(TestSubjectMap)을 입력으로 받습니다. 각 스킬은 자기 책임 범위만 보고, 모든 이력은 Ledger에 전수 보존되어 사람이 한 페이지로 검토할 수 있습니다.

llm-based-semantic-mutation-testing MUTATION · 민감도

AST 결정론 변이 + LLM 의미 변이로 원본을 흔든 뒤, 테스트가 잡는지(killed/survived/timeout/error) 측정.

  • Survivor마다 "왜 못 잡았나" 진단 → TestFinding
  • MutationLedger에 모든 변이(필터된 것·budget drop 포함) 전수 보존
  • 세션 내 캐시로 반복 비용을 ×Δ 수준으로 축소
deterministic-first budget-aware human-inspectable
agentic-adversarial-testing ADVERSARIAL · 견고성

정적 critique 4종과 동적 adversarial 5종을 결합해 테스트 자체의 약점·누락을 찾는다.

  • fast static gate로 약한 assertion·tautology·과결합·이름-행위 괴리 탐지
  • critique 신호를 힌트로 표적화된 edge case·boundary·state·fuzz·spec 공격
  • AdversarialLedger에 모든 공격 케이스를 verdict와 함께 보존
critique-first targeted-adversarial flakiness-gated

Skill A — llm-based-semantic-mutation-testing

테스트가 비즈니스 로직 변형을 잡아내는지 검증하고, 사람 검토용 ledger를 함께 산출합니다.

skills/llm-based-semantic-mutation-testing/SKILL.md META · Skill A
When to use
  • 변경(diff)이 비즈니스 로직 분기를 포함(if/switch, 가격 계산, 권한 체크, 상태 전이).
  • subject-under-test-locator가 primary_subjects.confidence ≥ 0.7인 테스트를 반환.
  • 정책에서 mutation: enabled (light · normal · max).
  • Adversarial critique 결과 "weak assertion·tautology가 적음" → 정량 신호로 보강이 필요할 때.

호출하지 말아야 할 경우: tier가 unit이 아닌 테스트, primary_subjects가 비어 있는 테스트, 어댑터 부재.

Input contract 
{
  "test_target": TestTarget,
  "test_subject_map": TestSubjectMap,
  "language_adapter": LanguageAdapter,
  "policy": {
    "mode": "light" | "normal" | "max",
    "budget": {
      "max_mutants": 200,
      "max_seconds": 600,
      "max_tokens": 200000

    "session": {

  }
}

Procedure (요약)
  1. 스코핑 — primary_subjects ∩ diff = 변이 대상 span 집합 S.
  2. 변이 생성 (병렬)
    • mutation-operator-syntactic — 어댑터의 AST 변이.
    • mutation-operator-semantic — LLM이 비즈니스 의도를 추론해 미묘하게 위반.
  3. 등가성 필터 — likely_equivalent는 실행 제외, ledger에는 보존.
  4. 주입·실행 (mutant마다) — 캐시 조회 → injector → runner → 롤백 → ledger.
  5. 진단 (survivor만) — "왜 못 잡았나" LLM 진단 → TestFinding.
  6. 집계 — score, operator별 분포, hotspot.
Ledger는 단일 sink가 아니라 각 단계가 부르는 기록기. 후보·필터된 변이·budget drop까지 모두 시간순으로 보존하여 사람이 "왜 빠졌나"까지 확인할 수 있어야 합니다.
Quality bar (불가침)
  1. 증거 인용 필수 — 모든 finding에 evidence.test_snippet, evidence.mutant_diff.
  2. syntactic이 잡은 것을 semantic이 중복 생성 금지.
  3. likely_equivalent는 점수 분모에서 제외.
  4. 각 단계의 토큰·시간을 ledger에 기록(ROI 분석용).
  5. 매 mutant 실행 직후 작업 트리 무결성 검증.
  6. 부분 결과의 정직성 — silent skip 금지.
  7. 캐시 무효화는 mutant span 단위로 보수적으로.

Skill B — agentic-adversarial-testing

테스트 자체를 공격해 약점과 누락을 드러내고, 모든 공격 케이스를 ledger에 보존합니다.

skills/agentic-adversarial-testing/SKILL.md META · Skill B
When to use
  • 모든 PR/세션의 fast static gate로 critique 계열을 먼저 돌릴 때.
  • Mutation 단계의 survivor 진단에서 "약한 assertion·누락 카테고리" 신호가 나와 표적화 보강이 필요할 때.
  • 정책에서 adversarial: enabled (light=critique only · normal · max).
  • 보안·결제·권한처럼 적대적 입력이 본질적으로 중요한 도메인.

Procedure — 두 트랙

Track A · 정적 critique (병렬, 결정론 우선)

  • test-assertion-analyzer — 약한 assertion, 부동소수점 ==, 누락된 negative.
  • test-tautology-detector — 항상 통과, 자기 검증.
  • test-coupling-analyzer — 과도한 mock, private API 접근, 구현 세부 의존.
  • test-behavior-naming-checker — 테스트 이름·docstring과 실제 검증 간 괴리(LLM 비교).

Track B · 동적 adversarial (critique 결과를 힌트로 표적화)

  • adversarial-edge-case-generator — null/empty/유니코드/타임존/오버플로우.
  • adversarial-boundary-explorer — 경계 상수 ±1, off-by-one.
  • adversarial-state-attacker — 호출 순서·재진입·idempotency 위반.
  • adversarial-input-fuzzer — hypothesis/fast-check 반례.
  • spec-divergence-prober — 주석·docstring vs 실제 동작 차이.

각 케이스는 verdict ∈ {caught_by_existing_test, uncaught, inconclusive}로 분류되어 ledger에 적재되며, uncaught만 finding 후보.

표적화 룰: Track A에서 "weak_assertion·tautology"가 나오면 edge_case·fuzzer +, "name_behavior_mismatch"가 나오면 spec_divergence ++, "over_coupling"이 나오면 state +.
Quality bar (불가침)
  1. 증거 인용 필수 — critique은 test_snippet, 동적은 case_repr.
  2. State attacker가 만든 비결정 케이스는 3회 연속 동일 verdict일 때만 finding 승격.
  3. Critique confidence 임계 미만은 severity=info 강등.
  4. 공격 케이스의 "적절성"도 ledger 항목 — 통과/실패 무관하게 전수 보존.
  5. Suggested fix는 추상 문구 금지. 입력값·기대값을 가진 코드 조각.
  6. Silent skip 금지 — degradations·notices에 사유 기록.
  7. Suppression 환류 준수 — 같은 false positive가 두 번 보이면 안 됨.
예시 — 약한 assertion + 누락된 경계 (Python · pytest)

대상 테스트:

def test_apply_discount_returns_a_number():
    assert apply_discount(User(tier="VIP"), 100) is not None

Track A — test-assertion-analyzer:

  • finding f_001, category=weak_assertion, confidence 0.93, severity major.
  • suggested_fix: assert apply_discount(User(tier='VIP'), 100) == 85.

Track B — adversarial-boundary-explorer (A의 신호로 가중치 +):

  • 케이스 c_017: (User(tier='VIP'), 0) → SUT가 음수 반환. 기존 테스트 못 잡음.
  • ledger verdict=uncaught, finding f_002, category=off_by_one, severity major.

두 스킬은 이렇게 묶입니다

최상위 오케스트레이터 test-validity-evaluator가 단일 진입점. 두 메소 스킬은 동일한 TestSubjectMap을 공통 입력으로 받고, 각자의 ledger를 산출한 뒤 validity-report-aggregator → html-report-renderer로 합쳐집니다.

입력: TestTarget, diff, policy │ ▼ [1a] test-tier-classifier → 비유닛 제외, scope_summary │ (included만 통과) ▼ [1b] subject-under-test-locator → TestSubjectMap ★ 공통 입력 │ ├─────────────────────────┬─────────────────────────┐ ▼ ▼ │ [2] llm-based-semantic- [3] agentic- │ (정책에 따라 mutation-testing adversarial-testing │ 병렬/순차) │ MutationReport │ RedTeamReport │ │ + MutationLedger │ + AdversarialLedger │ ▼ ▼ │ [4] validity-report-aggregator │ → ValidityReport (dedupe·우선순위·클러스터링) │ │ ▼ [5] html-report-renderer → report.html (자체 포함) │ ▼ 출력: ValidityReport (JSON) + report.html (사람용)
권장 내부 순서. ① Critique을 fast static gate로 먼저 → ② Mutation은 diff 주변만, syntactic→semantic → ③ Adversarial 동적은 survivor 진단 신호로 표적화 → ④ Aggregation & 렌더.

사용법

두 가지 시점의 사용법 — (A) 스킬을 워크스페이스에 두는 방법, (B) 오케스트레이터를 통해 호출하는 방법, (C) 결과를 읽고 환류하는 방법.

정책 파일 작성

두 스킬 모두 동일한 policy.session을 공유하고, 자기 트랙의 policy.{mutation,critique,adversarial}만 추가로 읽습니다.

B. 오케스트레이터를 통한 호출

상위 에이전트 입장에서 도구는 사실상 하나입니다.

tool: evaluate_test_validity

  호출 시점: 테스트를 작성/수정한 직후
  입력:
    - test_target: 변경된 소스·테스트 파일들
    - diff:        PR diff (옵션)
    - policy:      mode + budget + scope
  출력:
    - ValidityReport (JSON)
    - report.html   (자체 포함 단일 파일)
  후속 행동:
    severity ≥ major인 finding에 대해 테스트 보강 →
    재호출 (EvaluationSession이 recurrence_blocked로 무한루프 차단)

반복 평가 루프(evaluate → fix → re-evaluate)는 반드시 EvaluationSession을 통과합니다. 종료 조건:

  • target_reached — score ≥ target AND major+ == 0 (converged)
  • recurrence_blocked — 같은 finding_id가 fixed 주장 후 2회+ 재등장 (terminated)
  • max_iterations — 기본 5회 (terminated)
  • plateau — 연속 2회 major+ 감소 없음 AND score 개선 ≤ 0.5%p (terminated)
  • wontfix_only — 남은 finding 전부 suppressed (converged)
  • budget_exhausted — 토큰/시간 한도 (terminated)

C. 결과 읽기와 환류

report.html은 자체 포함 단일 파일이며 PR 코멘트·CI 아티팩트로 첨부 가능합니다. 사람 환류 버튼은 다음 세션의 동작을 바꿉니다.

  • [mark equivalent] → .testvalidity/equivalent_mutants.yml에 mutant_id 기록. 다음 세션의 generator가 재생성하지 않거나 filter가 force_equivalent로 분류.
  • [mark wontfix] → .testvalidity/suppressions.yml에 finding_id 기록. 표시되되 suppressed=true로 게이트.
  • [annotate fix idea] → 자유 텍스트가 ledger의 해당 레코드에 첨부.

D. 호출 시나리오 세 가지

작은 변경

critique 트랙 + mutation light. 토큰 절감, 빠른 신호 위주.

policy.mode = "light"
mutation.budget.max_mutants = 40
adversarial.enabled_generators = ["edge_case"]

복잡한 비즈니스 로직

semantic mutation + adversarial 강화.

policy.mode = "normal"
mutation.operators.semantic_enabled = true
adversarial.enabled_generators = ["edge_case","boundary",
  "state","spec_divergence"]

안전 민감 (결제·권한)

모든 트랙 max. budget을 크게.

policy.mode = "max"
mutation.budget.max_mutants = 500
adversarial.fuzz.max_examples = 1000
state.allow_nondeterminism = true

현재 상태와 한계

두 SKILL.md는 ADR-1·ADR-2 설계 충실도가 높은 명세 문서입니다. 다만 chrome-cdp처럼 신선한 세션의 Claude가 SKILL.md만 읽고 즉시 실행할 수 있는 단계까지는 도달하지 못했습니다. 솔직한 한계는 다음과 같습니다.

● Blocking — 아직 채워야 하는 것

1. description이 트리거가 아닌 기능 소개 형태. 호출 매칭이 약합니다.

2. 17개 primitive("mutation-operator-syntactic" 등)는 이름만 있고 실제 디렉터리·스크립트가 없습니다.

3. 핵심 컨트랙트(TestTarget, TestSubjectMap, LanguageAdapter 등)가 ADR-1 §4 / ADR-2 §3·§4 참조로만 존재합니다.
